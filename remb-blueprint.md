create extension if not exists vector;

### 1. users

```sql
create table users (
  id           uuid primary key default gen_random_uuid(),
  clerk_id     text unique not null,
  email        text unique not null,
  name         text,
  plan         text not null default 'free',  -- free | pro | team | enterprise
  created_at   timestamptz default now()
);

alter table users enable row level security;

create policy "users can read own row"
  on users for select
  using (clerk_id = auth.uid()::text);
```

---

### 2. projects

```sql
create table projects (
  id           uuid primary key default gen_random_uuid(),
  user_id      uuid not null references users(id) on delete cascade,
  name         text not null,
  description  text,
  repo_url     text,
  created_at   timestamptz default now()
);

alter table projects enable row level security;

create policy "users can manage own projects"
  on projects for all
  using (user_id = (select id from users where clerk_id = auth.uid()::text));
```

### 3. features

```sql
create table features (
  id           uuid primary key default gen_random_uuid(),
  project_id   uuid not null references projects(id) on delete cascade,
  name         text not null,          -- e.g. "oauth-login"
  description  text,
  status       text default 'active',  -- active | archived
  created_at   timestamptz default now(),
  updated_at   timestamptz default now()
);

alter table features enable row level security;

create policy "users can manage features of own projects"
  on features for all
  using (
    project_id in (
      select id from projects
      where user_id = (select id from users where clerk_id = auth.uid()::text)
    )
  );
```

---

### 4. context_entries

```sql
create table context_entries (
  id           uuid primary key default gen_random_uuid(),
  feature_id   uuid not null references features(id) on delete cascade,
  content      text not null,              -- the actual context (markdown)
  entry_type   text default 'manual',      -- manual | scan | decision | fix
  source       text,                       -- cli | ide | worker | web
  embedding    vector(1536),               -- openai text-embedding-3-small
  metadata     jsonb default '{}',         -- files touched, tags, branch, etc
  created_at   timestamptz default now()
);

alter table context_entries enable row level security;

create policy "users can manage context of own features"
  on context_entries for all
  using (
    feature_id in (
      select f.id from features f
      join projects p on f.project_id = p.id
      where p.user_id = (select id from users where clerk_id = auth.uid()::text)
    )
  );

-- Index for fast vector similarity search
create index on context_entries
  using ivfflat (embedding vector_cosine_ops)
  with (lists = 100);
```

### Semantic Search Function

```sql
create or replace function search_context(
  query_embedding vector(1536),
  project_uuid    uuid,
  match_count     int default 5
)
returns table (
  id          uuid,
  feature_id  uuid,
  content     text,
  entry_type  text,
  similarity  float
)
language sql stable
as $$
  select
    ce.id,
    ce.feature_id,
    ce.content,
    ce.entry_type,
    1 - (ce.embedding <=> query_embedding) as similarity
  from context_entries ce
  join features f on ce.feature_id = f.id
  where f.project_id = project_uuid
  order by ce.embedding <=> query_embedding
  limit match_count;
$$;
```

```

Step 8   Scaffold apps/worker — BullMQ worker connects to Upstash + Supabase
Step 9   Test the queue: enqueue a job from a server action, worker picks it up

Step 11  Build CLI package — save and get commands working against local dev server
Step 12  Build frontend pages — dashboard, feature detail, markdown view
Step 13  Build React Flow visualizer
Step 14  Add Realtime subscriptions — graph updates live when context saved
Step 15  Build MCP server package
Step 16  Deploy: Vercel (web), Railway (worker)
Step 17  Publish CLI + MCP to npm
```

## CLI Commands Reference

```bash
# Setup
ctx login                              # save API key to config
ctx whoami                             # verify auth

# Projects
ctx project create "my-app"            # create a project
ctx project list                       # list all projects
ctx project use "my-app"               # set active project

# Features
ctx feature create "oauth-login"       # create a feature
ctx feature list                       # list features in active project

# Context
ctx save "used PKCE flow, stored tokens in httpOnly cookies"
ctx save --feature "oauth-login" --type decision "chose NextAuth over custom"
ctx get --feature "oauth-login"        # returns formatted context block
ctx get --semantic "authentication"    # vector search across all features
ctx list                               # list all entries for active feature

# Scanning
ctx scan                               # scan current directory
ctx scan --dir ./src/auth              # scan specific folder

# Export
ctx export --feature "oauth-login"     # prints markdown
ctx export --feature "oauth-login" --out ./docs/oauth.md  # save to file
```

## MCP Server Behavior

When configured in an IDE, the MCP server exposes these tools to the AI:

| Tool | What it does |
|---|---|
| `get_context` | Fetches all context entries for a feature by name |
| `search_context` | Semantic search across all features in a project |
| `save_context` | Saves a new context entry (AI can self-document) |
| `list_features` | Lists all features in the current project |

**Claude Cursor config (`~/.cursor/mcp.json`):**
```json
{
  "mcpServers": {
    "remb": {
      "command": "npx",
      "args": ["remb-mcp"],
      "env": {
        "REMB_API_KEY": "ctk_live_xxxxx",
        "REMB_PROJECT": "my-app"
      }
    }
  }
}
```

At session start, the AI automatically fetches relevant context for the active project and injects it before the first message.

## Worker Job Flow

```
1. Server action calls enqueueScan(projectId, dir)
2. Job pushed to BullMQ queue on Upstash Redis
3. Railway worker picks up job
4. Worker reads file tree using glob
5. For each changed/relevant file:
   a. Read file content
   b. Send to OpenAI GPT-4o-mini with prompt:
      "Summarize this code. Identify: feature name, key decisions,
       dependencies, gotchas. Return JSON."
   c. Parse response
   d. Generate embedding with text-embedding-3-small
   e. Save context_entry to Supabase with embedding
6. Update scan_jobs row with status: done + result summary
7. Supabase Realtime fires → frontend updates
```

**Scanning prompt template:**
```
You are a senior developer documenting code for an AI context system.
Given the following code, extract:
- feature_name: what this code implements
- summary: 2-3 sentence plain English description
- key_decisions: array of architectural or implementation choices made
- dependencies: other features or libraries this relies on
- gotchas: things a developer must know when working with this code
- tags: relevant keywords

Return ONLY valid JSON. No preamble.

Code:
<code here>
```

### Optional Accelerators (if you want to move faster)

- **v0.dev** — generate Shadcn UI components from a description, then customize. Saves hours on boilerplate UI.
- **Supabase CLI** — run `supabase db diff` to auto-generate migration files from your local schema changes. Never write migrations by hand.
- **trigger.dev** — alternative to BullMQ + Railway for background jobs. Fully managed, great DX, has a free tier. Consider this if you want zero infra management for workers.
- **Inngest** — similar to trigger.dev, event-driven background jobs with a visual dashboard. Excellent for the scanning pipeline.
